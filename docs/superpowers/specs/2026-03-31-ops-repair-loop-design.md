# OpenClaw Command Deck 运维修复闭环设计

## 背景

当前项目已经具备较完整的本地运维面板能力：

- 可以读取本机 OpenClaw 的配置、计划任务、会话和诊断数据
- 可以执行部分控制动作，如切换模型、重启网关、运行任务、派发代理
- 可以将部分失败状态呈现为告警或页面摘要

但它仍然偏“看板”，而不是“问题处理工作台”。

现状的主要缺口不是缺少更多页面，而是缺少一条完整的问题处理链：

1. 识别问题
2. 判断根因
3. 生成修复方案
4. 执行低风险修复或引导高风险修复
5. 自动验证问题是否真的被修复
6. 留下可回看的处理记录

第一阶段需要把项目从“展示异常”推进到“围绕根因修复异常”。

## 目标

将 OpenClaw Command Deck 升级为带根因识别与修复后验证能力的运维工作台。

第一阶段目标：

- 对渠道/插件异常建立统一的根因识别模型
- 对模型/网关问题建立统一的根因识别模型
- 为代理/会话补充日志可见性，作为问题识别输入
- 将修复动作与问题根因绑定，而不是与页面按钮松散绑定
- 在每次修复动作后自动做验证，明确告诉用户问题是“已修复、部分修复还是未修复”

## 非目标

第一阶段不做以下内容：

- 黑盒式、多步全自动修复流程编排
- 复杂的规则 DSL 或通用工作流引擎
- 全量历史趋势分析平台
- 多租户、远程实例、集中托管模式

这些能力可以在问题模型稳定后再扩展。

## 用户价值

第一阶段完成后，用户面对常见异常时不再只看到“红点”或“失败”。

系统应能回答四个问题：

1. 发生了什么
2. 为什么发生
3. 现在最合适的修复动作是什么
4. 动作执行后到底有没有修好

这会显著降低以下操作成本：

- 从告警跳到控制台再找命令的切换成本
- 误把症状当原因的判断成本
- 修了但不知道是否真正恢复的验证成本
- 代理/会话相关问题没有日志入口时的排障成本

## 范围优先级

### 第一优先级：渠道与插件异常

覆盖问题类型：

- 渠道被关闭
- 插件被关闭
- 插件未安装
- 渠道与插件状态不一致
- 群策略或接入策略不安全
- 配置缺失导致渠道不可用

### 第二优先级：模型与网关问题

覆盖问题类型：

- 主模型缺失
- 主模型不可用
- 当前配置中存在可切换候选模型但未自动识别
- 网关不可达
- 网关需要重启才能恢复

### 第三优先级：代理与会话日志能力

覆盖问题类型：

- 会话或代理执行失败但当前页面缺少原始日志上下文
- 派发后无结果或疑似执行异常但无法快速定位
- 当前会话、代理页面只有摘要，没有“排障入口”

注意：第三优先级的核心不是先做更多“告警规则”，而是先把日志能力补齐，让后续根因判断更可信。

## 设计原则

### 1. 修复围绕根因，不围绕页面

不要把问题处理建模成“告警页的一个按钮”。

同一种表面异常可能有多个根因。例如“Discord 不可用”可能来自：

- 渠道开关关闭
- 插件开关关闭
- 插件未安装
- 配置缺失
- 群策略风险

系统必须先识别根因，再生成修复建议或动作。

### 2. 低风险动作可一键，高风险动作必须确认

动作必须分级：

- `auto`：低风险、可预测、可重复的修复
- `confirm`：有副作用、但仍适合由 UI 发起的修复
- `manual`：系统只能诊断，不能安全代替用户执行

### 3. 每次修复后必须验证

不允许出现“执行成功”但“问题是否解决未知”的状态。

系统需要明确输出：

- `resolved`
- `partially_resolved`
- `unresolved`

### 4. 代理/会话日志不是附属信息，而是排障输入

第三优先级不是单纯补一个日志页面，而是要把日志变成问题识别和修复验证的上游信号。

## 核心模型

第一阶段新增四层模型。

### 1. Signal Collectors

负责从不同数据源收集原始信号。

输入来源包括：

- `openclaw.json`
- `cron/jobs.json`
- `openclaw status --json`
- `openclaw logs --plain --limit 30`
- 会话存储
- 代理或工作区日志文件

输出是标准化信号，例如：

- 渠道启用状态
- 插件启用状态
- 插件安装状态
- 网关 reachability
- 主模型 key
- 模型候选列表
- 日志中的错误模式

### 2. Root Cause Classifiers

负责将原始信号归类为可处理的根因。

第一阶段建议至少支持这些根因枚举：

- `channel_disabled`
- `plugin_disabled`
- `plugin_missing`
- `channel_plugin_mismatch`
- `credential_missing`
- `unsafe_policy`
- `primary_model_missing`
- `primary_model_unavailable`
- `gateway_unreachable`
- `gateway_restart_required`
- `agent_dispatch_failure`
- `session_log_error_detected`

每个根因都需要包含：

- 分类类型
- 严重级别
- 证据摘要
- 证据详情
- 影响范围

### 3. Repair Plans

负责把根因映射成可执行修复方案。

每个修复方案包含：

- `repairability`: `auto | confirm | manual`
- `summary`
- `steps`
- `actions`
- `fallbackManualSteps`

其中 `actions` 可以是一个或多个受控动作定义，例如：

- 启用渠道
- 启用插件
- 切换主模型
- 重启网关
- 重新运行任务
- 修复投递目标

第一阶段不做复杂的多步自动化引擎，但允许一个 repair plan 包含“执行动作 + 验证动作”的结构。

### 4. Post-Repair Verification

负责在修复动作后重跑对应检查。

验证必须与根因一一对应，例如：

- `plugin_disabled` 修复后，验证插件状态是否为 enabled
- `channel_plugin_mismatch` 修复后，验证渠道和插件状态是否一致
- `primary_model_missing` 修复后，验证主模型是否存在且可解析
- `gateway_unreachable` 修复后，验证 gateway reachability 是否恢复
- `session_log_error_detected` 修复后，验证最近日志中是否仍出现相同错误模式

输出统一为：

- `resolved`
- `partially_resolved`
- `unresolved`

并附带验证证据。

## 页面与交互设计

第一阶段不新增大型独立子系统，优先在现有页面上形成闭环。

### 告警页

从“告警列表”升级为“问题列表”。

每个问题项展示：

- 问题标题
- 根因标签
- 严重级别
- 影响范围
- 推荐修复方式
- 可执行动作
- 最近一次验证结果

问题项展开后展示：

- 根因证据
- 修复步骤
- 修复后的验证详情

### 控制台

保留控制台作为动作执行中心，但动作来源不再只由静态表单决定。

控制台应能承接从问题页跳转过来的修复计划上下文，例如：

- 预选修复动作
- 显示动作风险级别
- 显示修复后将触发哪些验证

### 诊断页

从“状态摘要”扩展为“问题证据页面”。

重点展示：

- 网关状态证据
- 最近日志中的错误模式
- 安全审计摘要
- 与当前问题相关的底层信号

### 会话与代理页面

新增日志可见性入口。

第一阶段不追求复杂日志平台，但至少要做到：

- 可以看到最近错误片段
- 可以从问题项跳到相关代理或会话日志
- 可以看到日志错误模式是否仍在出现

## 动作分级

建议第一阶段按以下方式分级。

### Auto

可一键执行：

- 启用已安装但被禁用的插件
- 对齐渠道与插件启用状态
- 启用明确的渠道开关

### Confirm

执行前必须确认：

- 切换主模型
- 重启 gateway
- 重跑失败任务
- 修正任务投递目标

### Manual

仅展示建议与引导：

- 缺少外部平台凭据
- 外部平台侧配置错误
- 无法安全推断修复结果的问题

## 技术设计

### 服务端结构

第一阶段建议新增或扩展三类服务端模块：

- `lib/signals/*`
  负责采集原始信号
- `lib/root-causes/*`
  负责根因分类
- `lib/repair/*`
  负责修复计划与验证逻辑

现有 selector 可以继续负责页面视图模型，但“问题识别”逻辑应逐步从 selector 中抽离，避免视图模型承担诊断职责。

### API 设计

建议新增统一问题相关 API：

- `GET /api/issues`
  返回统一问题列表
- `POST /api/issues/:id/repair`
  执行修复动作
- `POST /api/issues/:id/verify`
  重跑验证

如果当前 App Router 结构不适合动态层级，可以采用等价的扁平路由形式。

### 操作留痕

每次修复与验证都应记录：

- 问题 ID
- 根因类型
- 执行动作
- 执行时间
- 执行结果
- 验证结果

第一阶段可以先复用现有 action history 能力，不要求独立持久化系统。

## 失败与回退策略

系统必须处理这些失败场景：

- 动作执行失败
- 动作执行成功但验证失败
- 动作执行成功但只部分恢复
- 验证本身无法完成

对于这些情况，UI 不能只显示“失败”，而需要明确区分：

- 动作失败
- 动作成功但问题未修复
- 动作成功但仍有残余风险

## 测试策略

第一阶段至少覆盖：

- 根因分类器单元测试
- 修复计划生成单元测试
- 修复后验证单元测试
- 关键问题页面与 API 的集成测试

夹具应继续使用仓库内固定测试数据，不依赖本机真实 OpenClaw 状态。

## 第一阶段验收标准

满足以下条件即可认为第一阶段达标：

1. 渠道/插件异常可被统一识别为明确根因
2. 模型/网关问题可被统一识别为明确根因
3. 至少一部分低风险问题支持一键修复
4. 至少一部分高风险问题支持确认后修复
5. 每次修复后系统会自动验证并展示结果
6. 代理/会话页面存在可用于排障的日志入口
7. 问题处理结果可以在 UI 中留痕回看

## 后续扩展方向

在第一阶段稳定后，可以继续演进：

- 多步自动修复剧本
- 问题优先级队列与 SLA
- 问题趋势与恢复时间统计
- 更细粒度的日志检索与过滤
- 远程实例支持
