# OpenClaw Command Deck 设计方案

## 1. 产品定义

`OpenClaw Command Deck` 不再只是一个读取本地状态的总览页，而是一个用来替代 OpenClaw 原生面板的主入口产品。

它的目标不是把所有能力平铺出来，而是做两层分工：

- 默认入口是 `高频工作台`
- 深层页是 `控制与管理中心`

这意味着它既要覆盖原生面板的核心能力，又要比原生面板更好用：

- 更少的认知切换
- 更短的操作路径
- 更强的建议能力
- 更清晰的高低频分层

## 2. 用户问题

这个产品首页必须优先回答三件事：

1. 我现在最该处理的 3-5 件事是什么
2. 哪些异常会影响今天的使用
3. 下一步最值得点哪个动作

只有在这三个问题被解决之后，才进入：

- 渠道管理
- 计划任务管理
- 代理管理
- 诊断和设置

## 3. 设计原则

### 3.1 高频优先

首页只服务高频任务：

- 重点事项
- 待处理
- 会话和执行推进
- 异常修复
- 快捷动作

### 3.2 低频下沉

以下内容从首页移走，统一下沉到二级页：

- 低频配置项
- 原始日志
- 长表格
- 技术术语堆叠
- 很少使用的复杂表单

### 3.3 先给动作，再给状态

所有异常和状态都应尽量带：

- 为什么要看
- 推荐动作
- 可直接执行的入口

纯展示而不能帮助行动的状态信息，不应占据首页核心区域。

### 3.4 技术状态翻译成工作语言

用户最终感知的是：

- 飞书能不能收发
- 计划任务会不会漏掉
- 哪个代理最近没有工作
- 哪个告警必须处理

所以页面应优先展示“对今天工作有什么影响”，而不是直接把底层字段暴露给用户。

## 4. 功能取舍

### 4.1 强化的能力

- 渠道健康与启停
- 计划任务启停、立即执行、失败修复
- 代理角色视角、模型切换、任务派发
- 告警到修复动作的闭环
- 会话与最近活跃任务
- 诊断和快速定位
- 快捷动作中心

### 4.2 弱化的能力

- 原始底层状态
- 全量日志浏览
- 低频配置表单
- 技术化字段过多的统计卡

### 4.3 移除或不作为重点的能力

- 技能市场入口
- 首页长表格
- 跟聊天窗口重复的大面积会话 UI
- 没有直接决策价值的“漂亮指标卡”

## 5. 信息架构

### 一级导航

- `工作台`
- `控制台`
- `消息与会话`
- `渠道`
- `计划任务`
- `代理`
- `告警`
- `诊断`
- `设置`

### 结构逻辑

- `工作台`
  - 高频入口
  - 今日重点与待处理
  - 系统异常和快捷动作

- `控制台`
  - 高价值动作集合
  - 常见修复入口
  - 最近操作

- `消息与会话`
  - 聚合最近会话
  - 看 agent 的真实工作流

- `渠道 / 计划任务 / 代理 / 告警`
  - 各自负责一个清晰控制面
  - 不再做大而全的混合页

- `诊断`
  - 原始状态快照
  - 日志和问题定位

- `设置`
  - 低频项
  - 面板偏好
  - 默认模型和配置入口

## 6. 首页工作台设计

### 6.1 页面目标

首页不是仪表盘，而是工作台。

它要把用户从“查看状态”直接推向“开始处理”。

### 6.2 首页模块

首屏建议分成五个区域：

1. `顶部状态条`
2. `今日重点`
3. `待处理`
4. `执行节奏`
5. `系统异常与快捷动作`

### 6.3 顶部状态条

只保留对决策有帮助的摘要：

- 当前实例
- 时间与时区
- Gateway 状态
- 在线渠道数
- 今日高优先级告警数
- 活跃代理数
- 当前主模型

右侧固定放高频动作：

- `发送任务`
- `立即执行任务`
- `查看告警`
- `进入控制台`

### 6.4 今日重点

只显示 3-5 张真正要看的卡片。

来源优先级：

1. 今天近期要处理的事项
2. 高优先级失败计划任务
3. 需要人工修复的异常
4. 高信号知识推送
5. 超时未处理的重要项

每张卡片字段：

- 标题
- 类型
- 来源
- 影响级别
- 时间上下文
- 一句解释
- 推荐动作

### 6.5 待处理

这是首页的主工作流区，不是技术清单。

待处理项来源：

- 最近会话
- 需要回复或跟进的任务
- 当前阻塞项
- 计划任务失败后的人工处理项

目标是让用户进来后立刻看到“现在能处理什么”。

### 6.6 执行节奏

用时间线表达：

- 今天的计划节点
- 即将执行的 cron
- 重要提醒
- 由 `chief-of-staff` 生成的节奏建议

### 6.7 系统异常与快捷动作

异常不再只显示报错文本，而是显示：

- 影响范围
- 是否阻断核心路径
- 推荐动作
- 修复按钮

快捷动作集中展示最常用的入口：

- 重载配置
- 重连渠道
- 立即执行计划任务
- 修复投递目标
- 给指定代理发任务

## 7. 二级页面设计

### 7.1 控制台

定位：把常用控制动作集中起来，避免用户在多个页面跳转。

内容：

- 计划任务高频操作
- 渠道高频操作
- 代理任务派发
- 模型切换
- 最近操作记录
- 常见问题修复入口

### 7.2 消息与会话

定位：看最近真实工作流，而不是看抽象状态。

内容：

- 最近会话列表
- 按 agent 聚合
- 最近消息时间
- 是否仍在活跃
- 关联入口

### 7.3 渠道

定位：渠道状态清晰、操作明确。

内容：

- 在线 / 离线状态
- 最近异常
- 启停
- 重连或重载
- 相关配置入口

### 7.4 计划任务

定位：计划任务的执行与修复中心。

内容：

- 启停
- 立即执行
- 下一次执行时间
- 最近结果
- 失败原因
- 修复 `delivery.to` 之类的高频问题

### 7.5 代理

定位：从“角色和职责”管理代理，而不是把代理当成技术节点。

内容：

- 主助手
- 执行系统
- 知识雷达
- 最近活跃
- 模型绑定
- 任务派发
- 工作区入口

### 7.6 告警

定位：只处理需要动作的告警。

内容：

- 严重级别
- 来源模块
- 影响描述
- 推荐动作
- 修复入口

### 7.7 诊断

定位：给高级用户和排障时使用。

内容：

- 状态快照
- 原始健康状态
- 最近日志
- 问题定位信息

### 7.8 设置

定位：承接低频、但仍然必要的配置项。

内容：

- 面板偏好
- 默认模型
- 默认入口
- 配置文件跳转
- 低频开关

## 8. 控制设计原则

### 8.1 按风险分层

- 一级动作：直接执行
  - 立即执行计划任务
  - 重载配置
  - 打开工作区

- 二级动作：确认后执行
  - 启停渠道
  - 启停计划任务
  - 切换默认模型

- 三级动作：预览后提交
  - 批量改配置
  - 批量调整规则

### 8.2 列表里不堆按钮

列表页只保留 1-3 个最合理动作，剩余动作收进详情或抽屉里。

目标是让页面更稳定，更像产品，而不是工具集合。

### 8.3 最近操作可追溯

所有高价值动作应能回看：

- 谁执行了什么
- 成功还是失败
- 最后反馈是什么

## 9. 视觉方向

风格不应像传统企业监控后台，也不应像花哨的营销页。

建议方向：

- 以浅暖色为底
- 使用更强的层次和留白
- 首页强调“今天该处理什么”
- 管理页强调“结构化与可控”
- 告警和风险使用更明确的层级色，而不是全站都靠彩色标签

## 10. 落地策略

### 第一阶段

先完成替代型骨架：

- 新导航
- 新工作台
- 新控制台
- 新消息与会话
- 新诊断
- 新设置

### 第二阶段

再增强：

- 更完整的控制流
- 最近操作记录
- 更好的异常修复引导
- 更细的角色工作流

### 第三阶段

最后做体验增强：

- 视觉收口
- 响应式优化
- 更强的空态和反馈
- 更顺滑的页面间联动

## 11. 结论

这个产品的核心不是“把 OpenClaw 的功能都网页化”，而是：

- 用一个更符合日常使用节奏的入口替代原生面板
- 把高频工作流前置
- 把低频复杂能力下沉
- 让每一个状态都更接近动作

真正的好用，不是功能更多，而是默认界面就知道用户下一步最该做什么。
